CLAUDE CODE: THE FULL COURSE 2026 - Master Claude Code Once and For All
6

Most developers use Claude Code like ChatGPT with a file system.
They type a request, get a file, feel productive.
They're using maybe 10% of what it can do.
This is the other 90%. 8 levels. The exact order. By the end you'll have a setup that makes one person work like a small team.

Level 1: Get Claude Code Running

Forget What You Think This Is
Most people open Claude Code and ask it questions.
That's not what it's for.
Claude Code is not a better ChatGPT. It's an agent that operates inside your machine. It reads your files. It writes code. It opens your browser, clicks elements, checks the console. It runs terminal commands. It deploys to production. It creates pull requests.
You don't ask it questions. You give it tasks and walk away.
ChatGPT gives you an answer. Claude Code changes your project.
The difference isn't the AI. It's what the AI can touch.
Install: Two Options
Option A: IDE Extension
Download Cursor or VS Code, open the Extensions panel, search "Claude", install Claude Code by Anthropic. Sign in. Done.
Good for getting started. Slightly less powerful than CLI.
Option B: CLI in Terminal
This is the one that unlocks everything.
Install Node.js first, then:
bash
npm install -g @anthropic/claude-code
Launch in any project folder:
bash
cd your-project-folder
claude
Subscriptions Plan: Pro / Max 5x / Max 20x Price: $20.mo / $100.mo / $200.mo When: Getting started / Serious work / Heavy parallel workloads
Quick math: the same token volume through the API costs ~$500/month.
The subscription is always the better deal. Not close.
Do This Before Anything Else
By default, Claude Code asks you to confirm every file operation. Claude wants to create a folder. Allow? [y/n]
If you say yes to every prompt, you're not delegating. You're supervising a very slow assistant.
Kill the confirmation loop permanently:
bash
claude --sandbox /path/to/safe/folder
Reload:
bash
source ~/.zshrc
Now launch with cc. No more prompts.
The course author ran this for six months on a production codebase. Zero problems. Claude is more careful with your files than you are. You write over important things all the time. Claude doesn't.
If you're still nervous, use sandbox mode for sensitive projects:
bash
claude --sandbox /path/to/safe/folder
Claude reads anywhere but only writes inside that folder.
Voice Input: The Multiplier Nobody Talks About
Here's what nobody tells you about working with Claude Code.
The bottleneck isn't Claude's speed. It's the quality of your instructions.
When you type, you abbreviate. You skip context. You assume Claude understands what you mean. It doesn't always. You get output that's 80% right, spend 20 minutes fixing the remaining 20%, and wonder why it's slower than just coding yourself.
When you dictate, something changes. Speaking is faster than typing, so you stop editing yourself. You explain the full picture. You describe edge cases. You mention constraints you'd normally skip. Claude gets better instructions and produces better output on the first try.
Wispr Flow is the best tool for this on Mac. Hold fn, speak anywhere in any application, it transcribes immediately. A prompt that takes 40 seconds to type takes 8 seconds to say. The transcription is accurate enough that you rarely correct anything.
Aqua Voice works similarly. Handy Computer is free.
Claude Code also has a built-in voice: hold Space in the terminal to dictate. Works, but external tools are sharper.
Try it for one session. You'll keep using it.
Commands Worth Knowing From Day One
bash
claude /cost        # How many tokens did this session cost?
claude /doctor      # Diagnose installation and config issues
claude /clear       # Start fresh session in same project
claude /memory      # See everything loaded into context right now
claude /status      # Current model, context %, active config
/cost is the one you'll use most. Run it after a heavy task and you'll immediately understand what's expensive and what isn't. Bloated CLAUDE.md? Shows up here. Unnecessary global tools? Shows up here.
/doctor before you spend 30 minutes debugging a broken MCP. It checks your Node version, auth status, MCP connections, known conflicts. Most Claude Code problems are environment problems. /doctor finds them in seconds.

Level 2: Make Claude Remember Your Project

The 33,000 Token Tax
Every new Claude Code session starts with zero knowledge of your project.
So the first thing it does is explore. It reads directories, opens files, figures out what's what.
On a small Chrome extension project, that exploration cost 33,000 tokens. The actual work in that same session cost 18,500 tokens.
The orientation cost almost twice the work.
And it happened again the next session. And the next.
This is a solved problem. Most people don't know it.
CLAUDE.md: Pay Once, Benefit Forever
Create a file called CLAUDE.md in your project root.
Claude reads it automatically at session start. Zero exploration needed.
Generate it automatically with:
bash
claude /init
Claude explores your project once, writes the file. You never pay for that exploration again.
Here's what a good CLAUDE.md looks like:
markdown
# Project: Markdown Checklist Chrome Extension

## What this is
Chrome extension that converts markdown files into interactive checklists.
Side panel in Chrome. IndexedDB for persistence.

## Stack
- Vanilla JS, no frameworks
- Chrome Extensions Manifest V3

## File structure
- /chrome/ — extension files
- /landing/ — marketing landing page

## Rules
- Never install npm packages without asking first
- Always English in code and comments
- When I say "deploy", run: cd landing && npx vercel --prod
Keep it under 500 lines. The goal is orientation, not documentation.
What belongs in CLAUDE.md: stack, file structure, rules, recurring commands.
What does NOT belong: Full API docs (Claude can look those up), explanations of how libraries work (waste of context), your README copy-pasted in, personal TODOs.
The test: if removing a line wouldn't change how Claude behaves, remove it.
Global Rules
Some rules should apply everywhere, not just one project.
Put those in:
markdown
~/.claude/CLAUDE.md
Rules that actually matter globally:
markdown
- Never install packages without permission
- Always English in code
- Never store API keys in code or local files
- Always use Git worktrees for parallel work
- Use Exa MCP for web search
- Use Context7 for any library or framework docs
Add rules during a session:
markdown
Add a global rule: never install packages on my machine without asking first
Takes effect next session. Every project.
Auto Memory: Useful But Not Trustworthy
Claude writes notes to itself during sessions. They live at:
markdown
~/.claude/projects/[project-name]/memory.md
What it actually writes looks like:
markdown
- Build script requires Node 20+, user's machine has 18. Use nvm first.
- Port 3000 is taken. Use 3001 for dev server.
- Playwright tests fail if Chrome is already open.
- User prefers pnpm over npm.
This is useful. Environment quirks, things that went wrong, machine-specific facts.
The problem: you can't see it accumulating. Claude sometimes writes things that are outdated, wrong, or just not useful. And those entries keep loading into context.
Rule: auto memory for environment facts. CLAUDE.md for everything that matters.
Check and clean it regularly:
markdown
claude /memory
This shows your global CLAUDE.md, project CLAUDE.md, and auto memory in one view. If you see something wrong, delete it.

Level 3: Give Claude the Right Tools

The Problem With Out-of-the-Box Claude Code
Out of the box, Claude Code reads files, writes files, and runs shell commands.
That's it.
Ask it to "open the landing page and check if the button alignment looks right." It can't. It would tell you to open it yourself.
Ask it to "fetch the latest Vercel docs and check if our deploy config is still correct." It can't. It would make something up from training data.
This is where most people stop. They get frustrated that Claude keeps hallucinating about library versions, can't see their browser, doesn't know about APIs released six months ago.
The fix is MCP. And skills. And Context7.
One by one.
MCP: Give Claude Eyes and Hands
MCP (Model Context Protocol) gives Claude callable tools with descriptions. When Claude understands what a tool does, it knows when to use it without being asked.
Install Chrome DevTools MCP globally, so it's available in every project:
bash
claude mcp add --scope user chrome-devtools npx @chrome-devtools/mcp@latest
Now Claude can open your browser, navigate to a URL, take screenshots, read the console, click elements, fill forms.
"Build the landing page and check how it looks in Chrome" becomes a real task, not a metaphor.
Install Exa for web search that actually works:
bash
# 1,000 free requests/month at exa.ai
claude mcp add exa "https://mcp.exa.ai/mcp?apiKey=YOUR_API_KEY_HERE"
Here's the difference between Exa and the built-in search.
Built-in search: "What are the latest Vercel CLI flags?" Returns three sentences scraped from a search result. Might be outdated. Definitely incomplete.
Exa: Returns the full Vercel documentation page. Current version. All flags. All examples. Claude reads the actual docs, not a summary of a summary.
For anything that requires real documentation, Exa is not optional.
View all installed MCPs:
bash
claude /mcp
One important thing on context: current Claude Code loads MCPs on-demand. When Claude decides it needs a tool, it loads the description. When it doesn't need it, nothing loads. Install globally only what you genuinely use in every project. Everything else belongs in project scope.
Skills: What MCP Isn't
MCP connects Claude to external services. Skills are something different.
A skill is a markdown file that lives inside your project:
markdown
.claude/skills/skill-name/SKILL.md
You write instructions. Claude reads them when it decides they're relevant. No external service, no API key, no dependency.
The key mechanism: only the skill header loads into context automatically. The full content loads only when Claude decides the skill applies to what you're doing.
So you can have 20 skills installed and pay almost nothing in context for them.
skills sh: The Community Registry
skills sh is a public registry of community-built skills. Think of it as npm but for Claude behaviors.
Browse, preview the actual skill content, install with one command:
bash
# Inside a running Claude Code session
claude /plugins
# Search by name
# Install → choose User (all projects) or Project (just this one)
Skills worth installing immediately:
frontend-design: Without this skill, Claude's UI output looks like AI-generated UI. Same grey cards. Same predictable layouts. Same Inter font at 16px. The skill teaches Claude to make actual design decisions. Install it globally. Every project benefits.
git-commit: Enforces conventional commits automatically. Every message follows the same schema. Semantic versioning stays consistent. Changelogs become auto-generated. The kind of thing you always meant to enforce but never did.
skill-creator: Helps you build your own skills. You describe what you want Claude to do, it scaffolds the SKILL.md. Meta but genuinely useful.
Skills that compound the most:
Library reference skills: Write one for every internal tool. Your internal billing library, your CLI wrapper, your database conventions. Claude will stop pattern-matching from generic examples and follow your actual patterns. One hour to write. Saves hours every week.
Verification skills: Describe exactly how to test that Claude's output works. Playwright steps. CLI assertions. State checks. The difference between "it looks right" and "it is right."
Runbook skills: Give Claude your debugging playbook. Service X shows symptom Y, check Z first. Claude follows it consistently. You stop re-explaining the process every time something breaks.
Create your own:
markdown
I want to create a skill that generates landing pages based on a reference screenshot.
The skill should:
- Accept a screenshot as input
- Use the structural layout but apply its own design
- Accept optional links for App Store, Google Play, Chrome Extension
- Skip buttons for missing links

Create this skill in .claude/skills/landing-creator/
Every hour you spend writing a good skill saves hours across every future session. They compound.
Context7: Stop Using Stale Docs
Claude's training has a cutoff date. React Router v7 released after it. Supabase changed their auth API. Vercel added new CLI flags.
Claude doesn't know any of that. It codes from memory. The memory is wrong.
Context7 fixes this. It fetches current documentation before Claude writes code.
bash
claude mcp add context7 npx @upstash/context7-mcp@latest
Add a global rule:
markdown
- Always use Context7 when working with any library, API, or framework
Now Claude pulls the actual current docs. Not its best guess from training data. The real thing, today.
Sub-agents: The Part That Changes Everything
Most people use Claude Code as one session, one task, one context window.
That's like running all your code in a single thread and wondering why it's slow.
Here's what one session looks like without sub-agents:
markdown
Task 1: Research architecture  →  100K tokens
Task 2: Plan implementation    →   50K tokens
Task 3: Build the feature      →  100K tokens
─────────────────────────────────────────────
75% of context used. Quality already degrading.
15 minutes elapsed. One task done.
Each sub-agent gets its own isolated context. It does one job and reports back. The main session stays clean.
And they run in parallel. Three tasks that take 15 minutes sequentially take 5 minutes.
Trigger on the fly:
markdown
Run two sub-agents in parallel.
Agent 1: dark minimal landing page design. Save to /landing-v3.
Agent 2: colorful modern landing page design. Save to /landing-v4.
Both use the frontend-design skill.
Watch them both run simultaneously while your main context barely moves.
View what they're actually doing:
bash
# Press Ctrl+O during a session
# Full tool call history, tokens per sub-agent, everything
Pick the right model per task. It matters.
Model: Opus / Sonnet / Haiku Use for: Planning, architecture, complex reasoning / Writing code, implementation / Searching docs, simple lookups
Opus on everything is like using a Formula 1 car to get groceries. Smart when it counts, overkill when it doesn't.
markdown
Run a Haiku sub-agent to search Context7 for React Router v7
nested routes documentation. Return only what's relevant.
Create a reusable sub-agent:
bash
claude /agents
# Create new agent → Scope: Project → Method: Create via Claude
Define it once:
markdown
Sub-agent "landing-creator":
- Loads landing-creator skill + frontend-design skill automatically
- Takes a style description as input
- Creates a new numbered variant based on /landing
- Outputs to /landing-v[n]/
Call it any time: Use the landing-creator sub-agent to build a version with dark theme and large typography.
Agent Teams: For When You Need a Debate
Agent Teams is experimental. Two agents that can actually talk to each other.
Enable in ~/.claude/settings.json:
json
{
  "experimental": {
    "agentTeams": true
  }
}
Sub-agents Agent Teams Communication Isolated Can debate each other Weight Lightweight Full sessions Best for Execution Research, decisions
Use case that works well:
markdown
Agent team. Two agents.
"Skeptic": every reason NOT to launch this extension publicly.
"Advocate": every reason to ship now.
Context: /landing/index.html
Three debate rounds. Then: final summary — weak points, strong points, what to fix.
Navigate between them with Shift + ↑/↓
Good for research and competing hypotheses. Not for writing code unless your process is already locked in. The complexity multiplies fast.

Level 4: The Context Window Is a Budget

What You're Actually Paying For
Every session has a context limit. Here's what fills it before you type a single word:
markdown
System prompt                ← always there
Built-in tools               ← always there
CLAUDE.md (project)          ← always there
CLAUDE.md (global)           ← always there
MCP tools (on-demand)        ← loads when needed
Skill headers                ← always there
Sub-agent definitions        ← always there
Auto memory                  ← always there
─────────────────────────────────────────────
Your conversation            ← grows as you work
Opus 4 gives you 1 million tokens. That sounds like a lot until you're 45 minutes into a complex task and Claude starts forgetting what you told it at the beginning.
Check what you've spent at any point:
bash
claude /cost
Run this after every major task for a week. You'll immediately understand which parts of your setup are expensive and which aren't. Bloated CLAUDE.md? Shows up here. Global tools you never use? Shows up here.
The Three Zones You Need to Know
Green zone (0-50%): Full quality. This is where you want to stay for every task.
Orange zone (50-70%): Claude starts losing track of things you said earlier. Instructions from the beginning of the session get deprioritized. You'll notice outputs that feel slightly off. Claude builds something that contradicts a constraint you mentioned 40 messages ago.
Red zone (70-85%): Noticeable degradation. Outputs become inconsistent. Claude starts treating your project like a generic project.
At 85%: Auto-compact triggers. Claude summarizes the conversation and continues. Sometimes it works. Sometimes critical context evaporates in the summary. You won't always know which.
The goal is simple: finish the task, start a new session. Don't carry old context into new work.
Install the Status Line
You need to see that percentage while you work.
bash
claude /status-line
Claude asks what to show. Recommended:
markdown
Current folder path, model name, context usage percentage. Minimal, no colors.
After setup:
markdown
~/projects/my-app  |  claude-opus-4  |  23%
When it crosses 50%, wrap up the current task. Don't start a new one in the same session.
Manual Compact: When You're In Too Deep
Already in the orange zone and not done?
bash
claude /compact "Focus on: the landing page redesign, current file structure,
the decision to use Manifest V3 over V2"
Claude compresses everything else, preserves what you specified. You lose the conversation history but keep the critical context.
More reliable than auto-compact. You choose what survives.
The Three Rules That Compound Over Time
One task, one session.
bash
claude /clear
Old context from a completed task doesn't help the next task. It uses space and introduces noise. Start clean.
Don't install things globally unless you need them everywhere.
Every global MCP and skill loads into every session's context. Install globally only what genuinely applies to every project. Everything else: project scope.
If a task is too big for one session, it's too big for one agent.
Break it into sub-tasks. Let sub-agents handle each in isolation. This is the architecture behind Level 7.

Level 5: Deploy

The Part Nobody Tells You
Building locally and running in production are two completely different disciplines.
For personal projects and internal tools: ship fast, iterate, learn. The cost of breaking something is low.
For public products that take payments or store user data: slow down and think.
SSL certificates, open ports, resource monitoring, backup strategies. And legal compliance: storing European user data without the right data processing agreements is a legal problem, not a technical footnote.
Before you deploy anything public:
markdown
What are the production risks for a public Chrome extension that stores user data?
What do I need to think about legally? Be specific.
Read what Claude returns. Actually read it, don't skim it.
Deploy to Vercel: Two Minutes to Live URL
For landing pages, frontend apps, static sites: Vercel is the fastest path from code to public URL.
Ask Claude to check options:
bash
How do I deploy /landing to Vercel without GitHub integration?
What are my options? Use Exa to fetch the current Vercel docs.
Claude reads the actual current docs and walks you through the choices. Then:
bash
npx vercel
Browser window opens for auth. After that, Claude configures everything. Live URL in about 2 minutes.
Subsequent deploys:
bash
npx vercel --prod
Your Secrets Are Not Secrets If They're in Code
People search GitHub for OPENAI_API_KEY every day. They find thousands of repos. Private repos get leaked. Repos get transferred. Developers forget what they pushed three months ago.
The rule is simple: production API keys never touch your codebase. Not in comments. Not in config files. Not in .env files that accidentally get committed.
Local development uses .env:
bash
# .env — never commit this file
OPENAI_API_KEY=sk-...
SUPABASE_URL=https://...
.gitignore must include:
bash
.env
.env.local
.env.production
Production uses GitHub Secrets. Covered in Level 6.

Level 6: Git, Worktrees, and Automation

Git Is Your Undo Button
Here's a scenario that happens to everyone eventually.
You're three sessions into a feature. Claude has been building steadily. Something looked off in session two but you kept going. In session three it becomes clear that an architectural decision from session one was wrong, and now it's embedded in 40 files.
With Git: you check out the session one commit. You're back to before the mistake. You fix the architecture. You continue.
Without Git: you either fix 40 files manually, or you start over.
bash
git init
git add .
git commit -m "initial commit"
Push to GitHub:
bash
gh auth login
gh repo create my-project --private
git push -u origin main
Commit often. Every meaningful change is a checkpoint. The cost is zero. The insurance value is enormous.
The Problem With Branches and the Fix
Branches are good for isolating work. But there's a catch.
Two agents working in the same folder on different branches will overwrite each other's files.
Claude opens popup.js. Second Claude opens popup.js. One wins. The other's work disappears.
Git worktrees solve this. Each worktree is a physical copy of the project in a separate folder, with its own branch:
bash
# -b creates the branch
git worktree add -b feature-1 ../my-project-feature-1
git worktree add -b feature-2 ../my-project-feature-2
Three folders now:
my-project/            ← stable main
my-project-feature-1/  ← agent 1 works here
my-project-feature-2/  ← agent 2 works here
Open two terminals. Two Claude Code sessions. Neither can touch the other's files.
bash
# Terminal 1 — cd my-project-feature-1 && cc
# "Build the dark variant of the landing page"

# Terminal 2 — cd my-project-feature-2 && cc
# "Build the colorful variant of the landing page"
Both run simultaneously. Both finish. You compare. You merge the one you want.
Cleanup when done:
bash
git checkout main
git merge feature-1
git worktree remove ../my-project-feature-1
git worktree remove ../my-project-feature-2
Automate Deploys: Never Push a Button Again
Ask Claude to set up automatic deployment once. After that, pushing to main is the only deploy step.
markdown
Create a GitHub Actions workflow that:
1. Triggers on push to main when /landing files change
2. Deploys to Vercel automatically
3. Uses Vercel credentials from GitHub Secrets
Claude creates .github/workflows/deploy.yml:
yaml
name: Deploy to Vercel
on:
  push:
    branches: [main]
    paths:
      - 'landing/**'
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Vercel
        env:
          VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
        run: |
          npm install -g vercel
          vercel --prod --token=$VERCEL_TOKEN
Every push to main: Vercel deploys. You don't touch it.
GitHub Secrets: The Right Way to Handle Credentials
Production keys go here. Never anywhere else.
GitHub: repo, Settings, Secrets and Variables, Actions, New repository secret.
What you need for Vercel:
markdown
VERCEL_TOKEN      ← Vercel account settings
VERCEL_ORG_ID     ← Vercel dashboard
VERCEL_PROJECT_ID ← Vercel dashboard
Can't find the IDs?
markdown
Find my Vercel org ID and project ID for /landing.
Check the .vercel config files or fetch from the Vercel API using Exa.
GitHub Variables GitHub Secrets Visible in logs Encrypted, never logged Safe for non-sensitive config API keys, passwords, tokens
Protect Main: Lock the Door
Once GitHub Actions is tested and confirmed working, protect the main branch.
GitHub: repo, Settings, Branches, Add branch ruleset, Default branch.
Turn on:
  • Require pull request before merging
  • Block force pushes
Claude now creates pull requests instead of pushing directly. Nothing merges to main without review.
Private repos need GitHub Pro ($4/month) for branch protection. Worth it.

Level 7: Build a Process That Actually Works

The Tool Is Not the Variable
DORA 2025 studied teams using AI at scale.
Strong teams got significantly stronger. Weak teams didn't. They used the same tools. The gap widened.
The difference wasn't the AI. It was the process around the AI.
A weak process with Claude Code produces output that's almost right and takes longer to fix than building correctly from scratch. A strong process produces shippable output consistently.
The process is learnable. It's not complicated. But you have to actually use it.
The Core Loop
markdown
Brainstorm → Spec → Plan → Execute → Review → Ship
Brainstorm is not planning. It's asking every question before writing a single line of code. What are the edge cases? What breaks? What did you forget to specify? This is where bugs get prevented, not fixed.
Spec is Claude writing down exactly what it understood. You read it. If something is wrong, you say so now. Changing requirements after sub-agents start executing wastes tokens, time, and patience.
Plan is decomposing the spec into discrete tasks. One task, one sub-agent, one isolated context. If a task requires the output of another task to complete, break it down further.
Execute is parallel. Each task runs in its own worktree. While one sub-agent builds the upload component, another builds the renderer. They don't interfere.
Review is a sub-agent checking each output against the spec. Not "does it look right." Does it meet the specified requirements, exactly.
Ship is a pull request. GitHub Actions handles the rest.
Superpowers: The Framework That Implements All of This
This workflow is what the Superpowers framework automates. obra/superpowers on GitHub, actively maintained, one of the most-used Claude Code skill sets.
Install:
bash

# Search: superpowers
# Install: User scope
Restart Claude after.
How It Runs in Practice
Brainstorm first:
markdown
I want to upgrade my Chrome extension:
- Upload any markdown file, convert it to a checklist
- Store multiple checklists, switch via dropdown
- Light and dark theme
- Notion-style design

Use the brainstorm skill.
Claude asks questions you didn't think to answer. Some of them will reveal assumptions you didn't know you were making. Answer all of them before moving on.
Spec:
Claude generates /docs/spec.md automatically after brainstorm.
Read the spec. This is the last cheap moment to change direction.
Plan:
markdown
Spec looks good. Create a plan. Save to /plans/
Output looks like:
markdown

- Create /chrome/upload.js
- Drag and drop + file picker
- Parse markdown into task array
- Store in IndexedDB

Task 2: Checklist renderer
- Update /chrome/popup.js
- Read from IndexedDB
- Checkbox state persistence
...
Execute:
markdown
Execute the plan with sub-agent driven development.
One sub-agent per task. Sonnet for implementation.
Code review after each task before proceeding.
Claude spawns agents. Each reads its task file, implements, gets reviewed, reports back.
Track what they're doing:
bash
# Ctrl+O during session
# Full tool call history, tokens per agent
Ship:
markdown
All tasks done. Create a pull request to merge into main.
GitHub Actions fires. Vercel deploys. Done.

What Actually Separates People
The developers who get the most out of Claude Code aren't the ones who know the most commands.
They're the ones who figured out that Claude Code is not a tool for executing tasks faster. It's a tool for operating at a different level of abstraction. You stop executing. You direct.
